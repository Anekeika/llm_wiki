# LLM Wiki vs RAG

**TLDR:** RAG обрабатывает знания при каждом запросе заново. LLM Wiki обрабатывает один раз при ingestion — и строит постоянную, накапливаемую базу.

## Ключевое различие

| Dimension | Traditional RAG | LLM Wiki |
|---|---|---|
| Обработка | При каждом запросе | Один раз при ingestion |
| Cross-references | Случайно / теряются | Предстроены, поддерживаются |
| Противоречия | Игнорируются | Фиксируются при ingestion |
| Накопление знаний | Нет — каждый запрос с нуля | Растёт с каждым источником |
| Формат | Временные ответы в чате | Постоянные Markdown-файлы |
| Владение данными | У провайдера | Локально у пользователя |

## Структура wiki

**Двухуровневая архитектура:**
- **Central Wiki** — глобальная база (UX-паттерны, сниппеты, стандарты)
- **Project Wiki** — специфичный контекст продукта (архитектура, логи, инциденты)

**Разделение raw и output — критически важно:**
Raw источники иммутабельны. AI пишет только в отдельную директорию. Если AI ошибся — удаляешь output, оригинал цел.

## Пять операций метаболизма памяти

1. **TRIAGE** — классифицировать тип источника (статья / лог / PDF / твит) перед извлечением. Разные типы требуют разной глубины.
2. **CONTEXTUALIZE** — интегрировать данные в существующие страницы, обновлять обратные ссылки. Один источник может обновить до 15 страниц.
3. **CONSOLIDATE** — синтезировать разрозненные заметки в Hub-страницы (MOC — Maps of Content).
4. **DECAY** — паттерн Actuel/Archive: актуальное в `## Actuel`, старое сжимается в одну строку в `## Archive`. Предотвращает превращение wiki в кладбище markdown.
5. **AUDIT** — еженедельная проверка на противоречия, orphan-страницы, устаревшие утверждения.

## Шаги построения wiki (по Карпати)

1. Собрать raw sources (статьи, книги, заметки, транскрипты)
2. Классифицировать перед ingestion (тип документа определяет глубину)
3. AI пишет wiki-страницы по шаблону: frontmatter + TLDR + тело + контраргументы/пробелы
4. Создать `index.md` — AI навигирует с него: читает TLDR, уходит вглубь только в релевантные страницы
5. Сохранять качественные Q&A как отдельные страницы с тегом `query-result`
6. Еженедельный lint-pass: противоречия, orphans, устаревшее

## Инструменты

- **Obsidian** — IDE для wiki:
  - Graph View: визуализация узлов знаний, orphan-страниц
  - Dataview plugin: SQL-подобные запросы по YAML frontmatter
  - Web Clipper: raw → Markdown с frontmatter в один клик
- **qmd** (Tobi Lutke) — поиск при >100 страниц: BM25 / semantic / hybrid + MCP-режим для Claude Code
- **Git** — version control: коммит после каждого ingest-сессии, rollback плохого прохода

## Практические инсайты (из Reddit, Outside_Brain1485, 1200+ заметок)

- Атомарные заметки (~40 строк, одна идея) точнее длинных wiki-страниц
- Ссылки деградируют при масштабировании → переходить на YAML frontmatter + typed entity references + grep вместо link traversal
- Длинные wiki-страницы снижают точность LLM — короткие атомарные лучше

## Критика и ограничения

- **Staleness**: LLM не знает, что факт изменился — нужны freshness metadata (version, timestamp, TTL)
- **Галлюцинации**: без AUDIT противоречия накапливаются
- **Scalability**: при >100 страниц нужен поисковый движок
- **Ценность learning**: автоматизация организации снижает глубину понимания (спорно)
